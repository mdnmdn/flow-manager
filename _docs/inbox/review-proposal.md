## Overview

This specification defines:

- Two document contracts used to pass context to an AI agent (`context.md`) and receive structured output back (`review.yaml` or `review.md`)
- The CLI surface for assembling context, invoking review, validating output, and applying changes to an Azure DevOps Pull Request
- The ADO REST API calls the CLI makes under the hood

The CLI does **no AI reasoning**. It assembles data, delegates reasoning to an external agent (Gemini CLI, Claude Code, etc.), validates the agent's output against a known schema, and executes deterministic ADO API calls.

PR updates are scoped to **comment threads only**. Work items and pipelines are out of scope for write operations.

## Document Formats

### `context.md` — Agent Input

Generated by `fm pr show`. This is the only input format — Markdown only, no YAML alternative. It is designed to be read naturally by an AI agent: prose sections for narrative content, structured sections for machine-checkable items.

The agent **must not be expected to parse** this document programmatically — it is prompt material.

```markdown
# PR #123 — Add payment retry logic

**Author:** marco.rossi
**Target branch:** main
**Source branch:** feature/payment-retry
**Created:** 2024-06-01T10:22:00Z
**Status:** Active

---

## Description

Implements exponential backoff for failed Stripe charges.
Retries are capped at 3 attempts with jitter.

### Open Points

- [ ] Handle partial refund edge case when charge partially succeeds
- [ ] Verify idempotency key is scoped per-user, not per-request
- [x] Add unit tests for retry logic

---

**File:** `src/payments/retry.ts` line 88
**Author:** elena.v · 2024-06-02T09:10:00Z

---

**Author:** marco.rossi · 2024-06-02T11:00:00Z

Added a TODO comment — will fix in a follow-up PR.

---

**Author:** elena.v · 2024-06-01T15:00:00Z

## Project Context

### README excerpt
> Payment service handles Stripe charge lifecycle. All charge operations
> must be idempotent. Retry logic must emit domain events on failure.

### AGENTS.md / CONTRIBUTING.md
> Code style: ESLint + Prettier enforced in CI.
> All public functions must have JSDoc.
> No hardcoded config values — use the `config` service.
```

**Rules for generation:**

- Threads are sorted: `active` first, then `resolved`
- Resolved threads are included but visually de-emphasised (for context on what was already addressed)
- Project context section is populated from `README.md`, `AGENTS.md`, `CONTRIBUTING.md` if present in the repo root — first 60 lines each, trimmed
- Open points are extracted from the PR description checklist syntax (`- [ ]` / `- [x]`)

### `review.yaml` — Agent Output (preferred)

The agent is prompted to produce this document. The CLI parses it deterministically. No prose interpretation occurs — every field maps directly to an ADO API call or is ignored.

```yaml
# review.yaml
# Generated by: gemini-cli / claude / codex
# PR: 123

summary: |
  Two of three open points are addressed in the diff. One major issue found
  around idempotency key scoping. Thread 42 is resolved by the code;
  thread 47 still requires action before merge.

recommendation: request_changes
# Allowed values: approve | request_changes | needs_discussion

threads:
  - id: 42
    action: resolve
    comment: |
      Config value now read from PAYMENT_MAX_RETRIES env var (retry.ts:91).
      Hardcoding removed — thread can be closed.

  - id: 47
    action: reply
    comment: |
      A TODO comment is not acceptable for merge. Please open a tracked
      work item and reference it here instead of leaving an inline TODO.

new_threads:
  - file: src/payments/stripe.ts
    line: 34
    severity: major
    comment: |
      Idempotency key uses `request_id` which is not scoped per-user.
      Two concurrent timeouts from different users could collide.
      Suggested fix: scope key as `{user_id}:{request_id}`.

  - file: src/payments/retry.ts
    line: 55
    severity: minor
    comment: |
      Public function `scheduleRetry` is missing JSDoc (required per CONTRIBUTING.md).

open_points:
  - ref: "Handle partial refund edge case"
    status: not_addressed
    comment: No changes relating to partial refunds found in the diff.

  - ref: "Verify idempotency key is scoped per-user"
    status: partially_addressed
    comment: |
      Issue identified in stripe.ts:34 — see new thread. Not yet fixed.

  - ref: "Add unit tests for retry logic"
    status: addressed
    comment: retry.test.ts adds 89 lines of coverage across all retry paths.
```

**Field constraints (enforced by `validate`):**

| Field | Type | Allowed values |
|---|---|---|
| `recommendation` | string | `approve`, `request_changes`, `needs_discussion` |
| `threads[].action` | string | `resolve`, `reply` |
| `threads[].id` | integer | Must exist in the PR |
| `new_threads[].severity` | string | `critical`, `major`, `minor`, `positive` |
| `new_threads[].file` | string | Must exist in the repo |
| `open_points[].status` | string | `addressed`, `not_addressed`, `partially_addressed` |
| `open_points[].ref` | string | Must match an item from `context.md` open points |

All `comment` fields are plain text, no Markdown rendering assumed.

### `review.md` — Agent Output (alternative)

Supported as input to `validate` and `apply` when `--format md` is specified. The CLI parses tagged fenced blocks embedded in the prose. All other Markdown content (prose summary, headings) is preserved as-is and posted as the top-level PR comment.

The agent is prompted to wrap every structured action in a tagged block. Prose outside blocks is ignored by the parser and used only for the summary comment.

````markdown
# Review — PR #123

Two of three open points addressed. One major issue found (idempotency key scoping).

```action:thread
id: 42
action: resolve
comment: Config value now read from PAYMENT_MAX_RETRIES. Hardcoding removed.
```

```action:thread
id: 47
action: reply
comment: TODO comment is not acceptable for merge. Open a tracked work item.
```

```action:new_thread
file: src/payments/stripe.ts
line: 34
severity: major
comment: Idempotency key not scoped per-user. Scope as {user_id}:{request_id}.
```

```action:open_point
ref: Handle partial refund edge case
status: not_addressed
comment: No changes relating to partial refunds found.
```

```action:open_point
ref: Add unit tests for retry logic
status: addressed
comment: retry.test.ts adds 89 lines of coverage.
```

**Recommendation:** request_changes
````

**Parser rules:**

- Only content inside ` ```action:<type> ``` ` blocks is parsed for structured actions
- Block content is parsed as inline YAML key-value pairs
- All field constraints from `review.yaml` apply identically
- The prose outside blocks is concatenated and posted as the overall PR summary comment
- `--format md` must be passed explicitly; YAML is the default

---

### JSON Schema for `review.yaml`

The CLI validates agent output against this schema before any write operation.

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["summary", "recommendation"],
  "additionalProperties": false,
  "properties": {
    "summary": { "type": "string", "minLength": 10 },
    "recommendation": {
      "type": "string",
      "enum": ["approve", "request_changes", "needs_discussion"]
    },
    "threads": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["id", "action", "comment"],
        "additionalProperties": false,
        "properties": {
          "id": { "type": "integer" },
          "action": { "type": "string", "enum": ["resolve", "reply"] },
          "comment": { "type": "string", "minLength": 1 }
        }
      }
    },
    "new_threads": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["file", "line", "severity", "comment"],
        "additionalProperties": false,
        "properties": {
          "file": { "type": "string" },
          "line": { "type": "integer", "minimum": 1 },
          "severity": {
            "type": "string",
            "enum": ["critical", "major", "minor", "positive"]
          },
          "comment": { "type": "string", "minLength": 1 }
        }
      }
    },
    "open_points": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["ref", "status", "comment"],
        "additionalProperties": false,
        "properties": {
          "ref": { "type": "string" },
          "status": {
            "type": "string",
            "enum": ["addressed", "not_addressed", "partially_addressed"]
          },
          "comment": { "type": "string" }
        }
      }
    }
  }
}
```

## CLI Commands

Global flags available on all commands:

```
--pr <id>         PR number. Defaults to the PR associated with the current branch.
--org <url>       ADO organisation URL. Defaults to ADO_ORG env var.
--project <name>  fm project. Defaults to ADO_PROJECT env var.
--repo <name>     Repository name. Defaults to ADO_REPO env var or git remote.
--token <pat>     ADO PAT. Defaults to ADO_TOKEN env var.
--output json     Machine-readable output for all read commands.
--no-color        Disable ANSI colour output.
```

### `fm pr show`

Assembles the full `context.md` document from ADO and the local repository.
**Read-only. No writes.**

```
fm pr show [<pr-id>] [--out <file>] [--no-project-context]

Options:
  --out <file>           Write to file instead of stdout. Default: stdout.
  --no-project-context   Skip injecting README / AGENTS.md / CONTRIBUTING.md.

Output: context.md (Markdown)
```

**Example:**

```bash
# Print to terminal
fm pr show

# Save to file for piping to agent
fm pr show --pr 123 --out context.md
```

**What it fetches:**

- PR metadata (title, description, author, branches, status) — `GET /pullRequests/{id}`
- All threads with status and comments — `GET /pullRequests/{id}/threads`
- Changed file list — `GET /pullRequests/{id}/iterations/{n}/changes`
- Project context files from repo root — local filesystem

### `fm pr thread list`

Lists comment threads on the PR.

```
fm pr thread list [<pr-id>] [--status active|resolved|all]

Options:
  --status <s>   Filter by thread status. Default: active.

Output: table (default) or JSON (--output json)
```

**Example output:**

```
ID   STATUS    FILE                          LINE  AUTHOR       PREVIEW
42   active    src/payments/retry.ts         88    elena.v      Max retries are hardcoded…
47   active    src/payments/retry.ts         102   marco.rossi  Added a TODO comment…
31   resolved  src/payments/stripe.ts        14    elena.v      Missing null check on…
```

**Example:**

```bash
fm pr thread list
fm pr thread list --status all
fm pr thread list --output json | jq '.[] | select(.status == "active")'
```

### `fm pr thread reply`

Posts a reply to an existing thread.

```
fm pr thread reply <thread-id> <message> [--pr <id>] [--resolve]

Arguments:
  thread-id   ID of the thread to reply to.
  message     Reply text. Use "-" to read from stdin.

Options:
  --resolve   Resolve the thread after posting the reply.
```

**Example:**

```bash
fm pr thread reply 42 "Fixed in latest commit."
fm pr thread reply 47 "Please open a work item for this." --resolve
echo "See commit a3f2c1" | fm pr thread reply 42 -
```

---

### `fm pr thread resolve`

Resolves one or more threads, with an optional closing comment.

```
fm pr thread resolve <thread-id> [<thread-id>...] [--pr <id>] [--comment <text>]

Arguments:
  thread-id   One or more thread IDs to resolve.

Options:
  --comment   Optional comment posted before resolving.
```

**Example:**

```bash
fm pr thread resolve 42
fm pr thread resolve 42 47 --comment "Addressed in this PR."
```

### `fm pr feedback validate`

Validates an agent-produced review file against the schema and cross-references it against live ADO data. **Read-only. No writes.**

```
fm pr feedback validate --file <path> [--pr <id>] [--format yaml|md]

Options:
  --file <path>    Path to review.yaml or review.md.
  --format yaml|md Explicit format. Auto-detected from extension if omitted.

Exit codes:
  0   Valid — no errors
  1   Hard errors — apply must not proceed
  2   Warnings only — apply can proceed with --force

Output:
```

**Example output:**

```
Validating review.yaml against PR #123…

Schema
  ✅ Valid YAML
  ✅ Required fields present (summary, recommendation)
  ✅ recommendation value: request_changes

Threads
  ✅ Thread 42 exists · status: active → action: resolve ✓
  ✅ Thread 47 exists · status: active → action: reply ✓
  ⚠️  Thread 31 exists but is already resolved → action will be skipped

New Threads
  ✅ src/payments/stripe.ts exists in repo
  ✅ src/payments/retry.ts exists in repo
  ⚠️  stripe.ts line 34 is outside the PR diff — will still be posted

Open Points
  ✅ "Handle partial refund edge case" matches context.md open point
  ✅ "Add unit tests for retry logic" matches context.md open point
  ❌ "Verify idempotency key is scoped per-user" not found in context.md open points

Result: 1 error, 2 warnings — fix errors before applying.
```

**Example:**

```bash
fm pr feedback validate --file review.yaml
fm pr feedback validate --file review.md --format md --pr 123
```

### `fm pr feedback apply`

Executes all actions from a validated review file against the fm pr. This is the only command that writes to ADO.

```
fm pr feedback apply --file <path> [--pr <id>] [--format yaml|md] 
                      [--dry-run] [--force]

Options:
  --file <path>    Path to review.yaml or review.md.
  --format yaml|md Explicit format. Auto-detected if omitted.
  --dry-run        Print every ADO API call that would be made, no writes.
  --force          Apply despite validation warnings (errors still block).

Exit codes:
  0   All actions applied successfully
  1   Validation failed — nothing written
  2   Partial failure — some ADO API calls failed (see output)
```

Runs `validate` internally before any write. Will abort on validation errors unless `--force` is passed (which only bypasses warnings, never hard errors).

**Execution order:**

1. Post overall summary as a new PR thread (from `summary` field)
2. Reply to existing threads (`threads[].action = reply`)
3. Resolve existing threads (`threads[].action = resolve`)
4. Post new file-level threads (`new_threads`)
5. Post open points summary as a reply on the summary thread

**Example output:**

```
Applying review to PR #123…

[1/5] Posting summary comment…              ✅ Thread 89 created
[2/5] Replying to thread 47…                ✅ Reply posted
[3/5] Resolving thread 42…                  ✅ Resolved
[4/5] Posting new thread on stripe.ts:34…   ✅ Thread 90 created
[5/5] Posting new thread on retry.ts:55…    ✅ Thread 91 created
      Open points summary…                  ✅ Posted as reply on thread 89

Done. 5 actions applied, 0 failed.
```

**Example:**

```bash
fm pr feedback apply --file review.yaml
fm pr feedback apply --file review.yaml --dry-run
fm pr feedback apply --file review.md --format md --force
```

## ADO REST API Reference

Base URL pattern: `https://dev.azure.com/{org}/{project}/_apis/git/repositories/{repo}`
API version: `7.1`

### Authentication

All requests use HTTP Basic auth with an empty username and a Personal Access Token (PAT) as the password, Base64 encoded.

```
Authorization: Basic <base64(":{PAT}")>
Content-Type: application/json
```

The PAT requires the following scopes:

| Scope | Required for |
|---|---|
| `vso.code` | Reading PR data, threads, file list |
| `vso.code_write` | Posting and updating threads |

---

### GET Pull Request

Fetches PR metadata: title, description, author, branches, status.

```
GET /pullRequests/{pullRequestId}?api-version=7.1
```

**Response fields used:**

```json
{
  "pullRequestId": 123,
  "title": "Add payment retry logic",
  "description": "Implements exponential backoff…\n- [ ] Handle partial refund…",
  "status": "active",
  "createdBy": { "displayName": "marco.rossi" },
  "sourceRefName": "refs/heads/feature/payment-retry",
  "targetRefName": "refs/heads/main",
  "creationDate": "2024-06-01T10:22:00Z"
}
```

### GET Pull Request Threads

Fetches all comment threads including status and nested comments.

```
GET /pullRequests/{pullRequestId}/threads?api-version=7.1
```

**Response fields used:**

```json
{
  "value": [
    {
      "id": 42,
      "status": "active",
      "threadContext": {
        "filePath": "src/payments/retry.ts",
        "rightFileStart": { "line": 88 },
        "rightFileEnd": { "line": 88 }
      },
      "comments": [
        {
          "id": 1,
          "parentCommentId": 0,
          "author": { "displayName": "elena.v" },
          "content": "Max retries are hardcoded…",
          "publishedDate": "2024-06-02T09:10:00Z",
          "commentType": "text"
        }
      ]
    }
  ]
}
```

**Thread status values:**

| Value | Meaning |
|---|---|
| `active` | Open, unresolved |
| `fixed` | Marked as fixed by author |
| `resolved` | Resolved by reviewer |
| `byDesign` | Won't fix by design |
| `closed` | Closed |
| `pending` | Pending response |

### GET Changed Files

Fetches the list of files changed in the latest PR iteration.

```
GET /pullRequests/{pullRequestId}/iterations?api-version=7.1
→ extract latest iteration id from response

GET /pullRequests/{pullRequestId}/iterations/{iterationId}/changes?api-version=7.1
```

**Response fields used:**

```json
{
  "changeEntries": [
    {
      "item": { "path": "/src/payments/retry.ts" },
      "changeType": "edit"
    }
  ]
}
```

`changeType` values: `add`, `edit`, `delete`, `rename`.

### POST New Thread

Creates a new comment thread, optionally anchored to a file and line.

```
POST /pullRequests/{pullRequestId}/threads?api-version=7.1
```

**Body — general PR comment (summary):**

```json
{
  "comments": [
    {
      "parentCommentId": 0,
      "content": "## 🤖 AI Review Summary\n\nTwo of three open points addressed…",
      "commentType": 1
    }
  ],
  "status": "active"
}
```

**Body — file-anchored thread:**

```json
{
  "comments": [
    {
      "parentCommentId": 0,
      "content": "Idempotency key not scoped per-user. Scope as {user_id}:{request_id}.",
      "commentType": 1
    }
  ],
  "status": "active",
  "threadContext": {
    "filePath": "src/payments/stripe.ts",
    "rightFileStart": { "line": 34, "offset": 1 },
    "rightFileEnd": { "line": 34, "offset": 1 }
  }
}
```

`commentType` values: `1` = text, `2` = codeChange, `3` = system.

**Response:** Full thread object with `id` of the created thread.

---

### POST Reply to Thread

Adds a reply comment to an existing thread.

```
POST /pullRequests/{pullRequestId}/threads/{threadId}/comments?api-version=7.1
```

**Body:**

```json
{
  "parentCommentId": 1,
  "content": "Fixed in commit a3f2c1.",
  "commentType": 1
}
```

`parentCommentId` is the `id` of the first comment in the thread (always `1` for the root comment).

---

### PATCH Thread Status

Updates the status of an existing thread (resolve, reactivate, etc.).

```
PATCH /pullRequests/{pullRequestId}/threads/{threadId}?api-version=7.1
```

**Body:**

```json
{
  "status": "fixed"
}
```

**Allowed status transitions from `active`:**

| Target status | When to use |
|---|---|
| `fixed` | Author marked their own issue as fixed |
| `resolved` | Reviewer confirmed the issue is resolved |
| `byDesign` | Intentional, not a bug |
| `closed` | Closing without resolution |

The CLI uses `fixed` when resolving threads on behalf of the review agent (it represents the author addressing feedback).

## End-to-End Flows

### Pipeline (automated, posts to PR)

```bash
fm pr show --pr $PR_ID --out context.md
gemini --yolo "$(cat .ado/prompts/review.md)" context.md > review.yaml
fm pr feedback validate --file review.yaml --pr $PR_ID
fm pr feedback apply --file review.yaml --pr $PR_ID
```

### Developer (local, inspect before posting)

```bash
# See what's on the PR
fm pr thread list

# Generate context and review locally
fm pr show --out context.md
claude context.md --output-format yaml > review.yaml

# Validate before touching anything
fm pr feedback validate --file review.yaml

# Dry-run to see exact API calls
fm pr feedback apply --file review.yaml --dry-run

# Post when satisfied
fm pr feedback apply --file review.yaml
```

### Partial manual workflow

```bash

# Inspect and hand-edit review.yaml, then validate
fm pr feedback validate --file review.yaml

# Resolve a specific thread manually
fm pr thread resolve 42 --comment "Confirmed fixed."

# Apply the rest from the file
fm pr feedback apply --file review.yaml
```
