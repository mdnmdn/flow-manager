# Configuration Structure

Flow Manager resolves configuration by merging sources in order of increasing precedence:

1. Built-in defaults
2. Config file (`fm`, `fm.toml`, `fm.yaml`, …)
3. `.env` file
4. Environment variables (`FM__` prefix)

Later sources override earlier ones, so environment variables always win.

---

## Initialising a config file

```sh
fm init                   # creates fm.toml in the current directory
fm init --path mydir/fm.toml  # custom path
```

`fm init` writes a commented template and then exits. It refuses to overwrite an existing file. Open the file and fill in your provider credentials before running any other command.

---

## Config file formats

The `config` crate (v0.15) is used for loading, and its default feature set supports all of the following. The filename **must** be `fm` (no extension) or `fm.<ext>`:

| Format | Extension | Notes |
|--------|-----------|-------|
| TOML   | `.toml`   | Default generated by `fm init` |
| YAML   | `.yaml` / `.yml` | |
| JSON   | `.json`   | |
| INI    | `.ini`    | Flat only — nested keys not supported |
| RON    | `.ron`    | Rusty Object Notation |
| JSON5  | `.json5`  | JSON with comments |

When you call `File::with_name("fm")`, the crate probes for each supported extension in turn. Use any format you prefer; the field names and nesting are the same regardless.

## Full config reference

### `[provider]` — provider selection (required)

Exactly one provider block must be present. Use the `type` key to select which one.

#### Azure DevOps

```toml
[provider]
type = "ado"

[provider.ado]
url     = "https://dev.azure.com/your-org"
project = "your-project"
pat     = "YOUR_PAT_HERE"
```

| Field     | Type   | Description |
|-----------|--------|-------------|
| `url`     | string | Organisation URL |
| `project` | string | ADO project name |
| `pat`     | string | Personal Access Token |

#### GitHub

```toml
[provider]
type = "github"

[provider.github]
token    = "YOUR_TOKEN_HERE"
owner    = "your-org"
repo     = "your-repo"
base_url = "https://api.github.com"   # optional, for GitHub Enterprise
```

| Field      | Type   | Description |
|------------|--------|-------------|
| `token`    | string | Personal Access Token or App token |
| `owner`    | string | Organisation or user name |
| `repo`     | string | Repository name |
| `base_url` | string | Override for GitHub Enterprise (optional) |

#### GitLab

```toml
[provider]
type = "gitlab"

[provider.gitlab]
token      = "YOUR_TOKEN_HERE"
namespace  = "your-group/your-project"
project_id = 12345          # optional, speeds up API calls
base_url   = "https://gitlab.com"  # optional, for self-hosted
```

| Field        | Type    | Description |
|--------------|---------|-------------|
| `token`      | string  | Personal or Project Access Token |
| `namespace`  | string  | Group/project path |
| `project_id` | integer | Numeric project ID (optional) |
| `base_url`   | string  | Override for self-hosted GitLab (optional) |

### `[fm]` — workflow defaults (optional)

```toml
[fm]
merge_strategy = "squash"      # squash | merge | rebase
default_target = "main"        # baseline branch name
default_wi_type = "User Story" # work-item type for `fm work new`
submodules = ["_docs"]         # submodules managed transparently
```

| Field            | Default        | Description |
|------------------|----------------|-------------|
| `merge_strategy` | `"squash"`     | PR merge strategy |
| `default_target` | `"main"`       | Default baseline branch |
| `default_wi_type`| `"User Story"` | Work-item type created by `fm work new` |
| `submodules`     | `[]`           | Submodule paths handled by `fm commit` / `fm push` |

All fields are optional — built-in defaults are used when absent.

---

### `[sonar]` — SonarQube (optional)

```toml
[sonar]
url      = "https://sonar.example.com"
token    = "YOUR_SONAR_TOKEN_HERE"
projects = ["my-project-1", "my-project-2"]
```

| Field     | Type     | Description |
|-----------|----------|-------------|
| `url`     | string   | SonarQube server URL |
| `token`   | string   | User or project token |
| `projects`| string[] | Default projects for `sonar issues` (optional) |

---

### `[ado]` — legacy ADO block (deprecated)

A top-level `[ado]` block is still accepted for backward compatibility and is automatically promoted to `[provider]` at load time. Prefer `[provider]` for new configs.

## Environment variables

Every config key can be set via an environment variable. The mapping rule is:

```
FM__<SECTION>__<KEY>
```

Double underscores (`__`) separate nesting levels. Examples:

| Config key              | Environment variable          |
|-------------------------|-------------------------------|
| `provider.ado.url`      | `FM__PROVIDER__ADO__URL`      |
| `provider.ado.pat`      | `FM__PROVIDER__ADO__PAT`      |
| `provider.github.token` | `FM__PROVIDER__GITHUB__TOKEN` |
| `fm.default_target`     | `FM__FM__DEFAULT_TARGET`      |
| `sonar.token`           | `FM__SONAR__TOKEN`            |

Environment variables are read after the config file, so they can override any file value without modifying it. This is the recommended approach for secrets in CI/CD pipelines.

### `.env` file

A `.env` file in the working directory is loaded automatically (via `dotenvy`) before environment variable resolution. It follows the standard `KEY=VALUE` format:

```sh
FM__PROVIDER__ADO__PAT=my-secret-pat
FM__SONAR__TOKEN=my-sonar-token
```

Add `.env` to `.gitignore` to keep secrets out of version control.
